# 網站架構

> 最後更新:與 `內容規劃.md` 並列維護

## 概覽

本專案使用 Docker 容器化部署,目前由七個服務組成,涵蓋內容站、選物商城、後台、巨集儀表板與資料排程。所有 Flask 服務都透過 gunicorn 提供,並由 Nginx 做路徑基底的反向代理。

```
                     ┌─────────────┐
     使用者 ────────► │   Nginx    │ (port 80)
                     │ 反向代理     │
                     └──────┬──────┘
                            │
       ┌────────────┬───────┼───────────┬────────────┐
       │            │       │           │            │
       ▼            ▼       ▼           ▼            ▼
   ┌────────┐  ┌────────┐ ┌────────┐ ┌────────┐  ┌────────┐
   │  Blog  │  │  Shop  │ │ Admin  │ │Dashboard│  │   DB   │
   │ Flask  │  │ Flask  │ │ Flask  │ │ Flask   │  │  MSSQL │
   │ :5000  │  │ :5001  │ │ :5002  │ │ :5003   │  │ :1433  │
   └────────┘  └────────┘ └────────┘ └────┬────┘  └────────┘
                                           │ ▲
                                           │ │
                              ┌────────────┘ │
                              │ 排程 (cron-like) │
                              │ scheduler.py   │
                              └────────────────┘
```

Dashboard 額外有一個 `dashboard-scheduler` 容器,負責定期呼叫各 provider 抓公開資料。

## Container 規劃

| Container              | 用途                                  | 對外 Port | 內部 Port |
|------------------------|---------------------------------------|-----------|-----------|
| `nginx`                | 反向代理,路徑基底分流                  | `80:80`   | 80        |
| `blog`                 | 部落格 Flask 應用                      | `5000:5000` | 5000    |
| `shop`                 | 選物商城 Flask 應用                    | `5001:5001` | 5001    |
| `admin`                | 後台 Flask 應用                       | `5002:5002` | 5002    |
| `dashboard`            | 巨集儀表板 Flask 應用 + 頁面           | `5003:5003` | 5003    |
| `dashboard-scheduler`  | 排程 worker(同一 image, 跑 `scheduler.py`) | -     | -        |
| `db`                   | SQL Server 2022 Express                 | `1433:1433` | 1433  |

`blog` / `shop` / `admin` / `dashboard` 都同時使用 `expose` 與 `ports` 對外,目的是開發期能直接 `curl http://localhost:<port>/` 略過 Nginx 驗證,正式部署時要記得拿掉 `ports:` 段落。

## Nginx URL 路由(路徑基底)

| URL 路徑       | 目的地       | 說明 |
|----------------|--------------|------|
| `/`            | 301 → `/blog/` | 根目錄導向部落格 |
| `/blog/*`      | `blog:5000`  | 部落格所有請求,會去除 `/blog` 前綴後轉送 |
| `/shop/*`      | `shop:5001`  | 商城所有請求,會去除 `/shop` 前綴後轉送 |
| `/admin/*`     | `admin:5002` | 後台所有請求 |
| `/dashboard` 與 `/dashboard/*` | `dashboard:5003` | 儀表板頁面與 API |
| `/health`      | (Nginx 內部) | 健康檢查端點 |

## 環境變數

由 `.env` 帶入,範例請見 `.env-sample`:

| 變數                    | 用途 |
|-------------------------|------|
| `SA_PASSWORD`           | SQL Server sa 密碼 |
| `SESSION_SECRET`        | Flask session 簽章密鑰 |
| `FRED_API_KEY`          | FRED(聯準會經濟資料)API key |
| `DASHBOARD_INGEST_TOKEN`| Dashboard 手動 ingest 端點的 bearer token(留白 = 開發期免 token) |

`FRED_API_KEY` 也會透過 `docker-compose.yml` 注入到 `dashboard-scheduler`,沒有設定時 provider 仍可運作(會回傳錯誤但不中斷其他 job)。

## Flask 服務路由

### Blog(`blog:5000`)

| 路由 | 說明 |
|------|------|
| `GET  /`                       | 取得 API 版本資訊 |
| `GET  /posts`                  | 取得所有文章 |
| `GET  /posts/<id>`             | 取得指定文章 |
| `GET  /health`                 | 健康檢查 |
| `GET/POST /categories` 等      | 詳細 CRUD 見 `內容規劃.md` |

### Shop(`shop:5001`)

| 路由 | 說明 |
|------|------|
| `GET  /`                       | 取得 API 版本資訊 |
| `GET  /products`               | 取得所有商品 |
| `GET  /products/<id>`          | 取得指定商品 |
| `GET  /orders`                 | 取得所有訂單 |
| `GET  /health`                 | 健康檢查 |

### Admin(`admin:5002`)

| 路由 | 說明 |
|------|------|
| 後台登入與 CRUD 介面       | 由後台前端模板提供 |
| `EXPECTED_USER_TYPE=admin`  | 透過 `SESSION_COOKIE_DOMAIN=.localhost` 共享 session |

### Dashboard(`dashboard:5003`)

頁面:

| 路由 | 說明 |
|------|------|
| `GET /`            | 巨集儀表板首頁(分頁 tab + 冷熱判讀) |
| `GET /scheduler`   | 排程狀態監控頁 |

API:

| Method | Path | 說明 |
|--------|------|------|
| GET    | `/health`                         | 健康檢查 |
| GET    | `/api/summary`                    | 儀表板單層摘要(items + 冷熱 summary) |
| GET    | `/api/agent/dashboard`            | **AI Agent 一次讀所有指標的端點**:回傳 `categories`、`items`、`temperature_summary` |
| GET    | `/api/txf_price`                  | 取得台指期近月前一交易日收盤 |
| POST   | `/api/ingest/<code>`              | 手動餵入任何序列的觀測值,需 `DASHBOARD_INGEST_TOKEN`(或留白) |

## 排程 (`dashboard-scheduler`)

`dashboard/scheduler.py` 為獨立 worker,與 `dashboard` 容器共用 image,只差在 `command: ["python", "scheduler.py"]`。

- 每 `DASHBOARD_SCHEDULER_POLL_SECONDS` 秒(預設 30)輪詢 `economic_fetch_jobs`
- 透過 `providers/__init__.py` 的 registry 取得對應 provider:
  - `fred` — 利率、CPI、PCE、NFP、GDP、SP500、NASDAQ、VIX
  - `twse_openapi` — 台股加權、外資買賣超、融資餘額
  - `taifex` — 台指期近月前一交易日收盤
  - `cboe` — Put/Call Ratio
  - `cnn_fear_greed` — Fear & Greed Index
  - `yfinance` — 黃金 / 白銀現貨(目前有 Yahoo rate limit 風險,需監控)
  - `derived` — 2Y-10Y 利差
  - `taiwan_economic` — 台灣央行政策利率 / CPI / GDP / 失業率(目前為手動 ingest fallback)
- 觀測值寫入 `economic_observations`,並在 `economic_fetch_runs` 留下成功/失敗紀錄
- 失敗時 status 為 `error`,message 寫入最近 480 字錯誤訊息,可在 `/scheduler` 頁面看到

## 資料庫

- **主機**: `db` 容器
- **連接埠**: `1433`
- **帳號**: `sa`
- **密碼**: 從 `.env` 的 `SA_PASSWORD` 帶入
- **資料庫名稱**: `BlogShopDB`
- **初始化**: `init-scripts/init.sql` 第一次啟動時自動執行(由 MSSQL image 機制)

Schema 細節見 `內容規劃.md` 的「Schema 規劃」段落。

## 開發期常用指令

```bash
# 啟動整組服務
docker compose up -d --build

# 重新建立並重啟 dashboard / scheduler
docker compose up -d --build --force-recreate dashboard dashboard-scheduler

# 直接打 API / 略過 Nginx
curl http://localhost:5000/   # blog
curl http://localhost:5001/   # shop
curl http://localhost:5002/   # admin
curl http://localhost:5003/   # dashboard
curl http://localhost:5003/scheduler   # 排程狀態頁
curl http://localhost:5003/api/agent/dashboard | jq .   # AI Agent 端點

# 觸發所有排程立即跑一次
docker compose exec -T dashboard python -c "from datetime import datetime; from app import app, db, EconomicFetchJob; \
ctx=app.app_context(); ctx.push(); \
EconomicFetchJob.query.update({'next_run_at': datetime.utcnow()}, synchronize_session=False); \
db.session.commit()"

# 觀察排程 log
docker compose logs --tail=200 dashboard-scheduler
```

## 部署注意事項

- 上正式環境時,請把 `docker-compose.yml` 中下列服務的 `ports:` 段落拿掉,只保留 `expose:`,並只透過 Nginx 對外:
  - `blog`、`shop`、`admin`、`dashboard`
- `SA_PASSWORD`、`SESSION_SECRET` 在正式環境請用 secrets manager
- `FRED_API_KEY` 為付費/有 rate limit 的 key,正式部署前請評估流量
- 預期監控重點:
  - `dashboard-scheduler` log 中 `job XXX failed`
  - `/scheduler` 頁面 status = `error` 的排程數
  - Yahoo `yfinance` rate limit(目前每日觀察仍偶發)
